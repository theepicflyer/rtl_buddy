# audit-docs

You are reviewing the `docs/` directory of the rtl_buddy repo — the MkDocs site served at <https://rtl-buddy.github.io/rtl_buddy/>.

## Page types and their standards

The docs have two distinct types of pages with different standards:

**Concept and guide pages** (`concepts/`, `quickstart`, `install`, `agents`, `migrations`) — these are explanatory. A reader arrives without knowing how the feature works. Each page must:
- Open with a plain-English explanation of what the feature is and *when and why* to use it
- Build understanding progressively — motivation first, mechanics second
- Use section headings to let a reader scan and jump to what they need
- Include runnable examples for every non-trivial pattern
- Stay focused — one concept per page, no padding, no field enumeration without prose around it

**Reference pages** (`reference/cli.md`, `reference/yaml.md`) — these are intentionally exhaustive. Field tables, flag descriptions, and schema examples are appropriate here. The standard for reference pages is *accuracy and completeness*, not narrative flow. `reference/cli.md` is auto-generated by `python scripts/gen_cli_reference.py` and must not be edited by hand.

Apply the wrong standard to the wrong page type and you will get bad audit results.

## How to audit

### For a general review

1. List all pages: `rb docs list`.
2. Classify each page as concept/guide or reference.
3. Evaluate concept/guide pages for explanation quality; evaluate reference pages for accuracy and completeness.
4. Cross-check `reference/cli.md` against `rb --help` and per-command `--help` output for drift.
5. Cross-check `reference/yaml.md` against `src/rtl_buddy/config/` for undocumented fields.

### For a targeted review (new feature)

1. Look at recent commits in `rtl_buddy/` to identify what was added.
2. Check whether a concept page covers it with explanation. If not, flag as missing.
3. Check that `reference/yaml.md` documents any new YAML fields and `reference/cli.md` reflects any new flags (re-run `python scripts/gen_cli_reference.py` if needed).

## Evaluation criteria

**Concept/guide pages:**

| Criterion | Standard |
|-----------|----------|
| `description:` frontmatter | Required — CI enforces this |
| Opening paragraph | Answers "what is this and when do I use it" before any mechanics |
| Section structure | `##` per major concept; scannable without reading in full |
| Runnable examples | Every non-trivial config or command shown copy-pasteable |
| Cross-references | Links to reference pages for schema details rather than restating them |
| No unexplained enumerations | Field lists must have surrounding prose explaining intent |

**Reference pages:**

| Criterion | Standard |
|-----------|----------|
| `description:` frontmatter | Required — CI enforces this |
| `reference/cli.md` | Must match current CLI exactly — regenerate if any drift |
| `reference/yaml.md` | Every config field documented; required vs optional clearly marked |
| Examples | At least one complete working example per major config block |

## Output format

For a general review: one table per page listing `Page | Type | Criterion | Status | Note`. For a targeted review: a short list of gaps with one recommended action each. Flag any missing `description:` frontmatter as a CI-blocking issue.
